# action pipeline behavior walk-through (two clamps / raw gaussian vs tanh / delta integrator reset / single-vector two-dynamics / arm PD gain DR)

Conversational code walk-through of the envs/main action pipeline (2026-07-02, NO code change). Answers a series of Q&A on the action path documented in docs/reference/action-pipeline.ko.md. Records how the existing code behaves and WHY, so future work knows what rests on standard practice vs project choice. Companion to the policy-head card [[policy_head_design_rationale_and_literature_standing_encoder_inp]] (network-internal std/entropy) and the thruster-curve card [[thruster_nonlinear_curve_t200_sim_to_real_off_by_default_deadban]].

TWO ACTION CLAMPS, and CURRENT cfg state. The [-1,1] bound is enforced by TWO independent clamps (action-pipeline.md sect.3). Clamp #1 = RslRlVecEnvWrapper.step() (isaaclab vecenv_wrapper.py:153-154) torch.clamp(actions, -clip_actions, +clip_actions), OPTIONAL: only fires when clip_actions is not None. Clamp #2 = ALBCEnv._update_action_buffers() (albc_env.py:452) self._actions.clamp(-1,1), ALWAYS (unconditional), the canonical one. CURRENT STATE: Clamp #1 is a NO-OP. rsl_rl_ppo_cfg.py does NOT set clip_actions, so it inherits the isaaclab base default clip_actions=None (rl_cfg.py:181). train.py:292 passes agent_cfg.clip_actions(=None) to the wrapper -> vecenv guard skips it. train_student.py:120 explicitly passes clip_actions=None too. So only Clamp #2 actually bounds actions in every path. Redundancy is by design (sect.8), not a bug; env clamp is canonical, wrapper clamp is a library-level guard that is off here. Consequence: the entropy-collapse / boundary-mass-waste implications of external clamp all come from Clamp #2.

RAW GAUSSIAN SAMPLE + WHY EXTERNAL CLAMP (not tanh). Policy outputs a ~ N(mu(o), sigma) sample with NO squashing and NO clamping at the policy stage (act() samples, act_inference() returns mean; both raw). "raw" = unbounded Normal, so large sigma puts probability mass outside [-1,1]. The bound is an env-side INTERPRETATION CONTRACT, not policy squashing (PPO/TRPO standard). Difference from SAC tanh: (a) external clamp -> gradient is 0 beyond the boundary (hard saturation), probability mass wasted outside [-1,1] -> a structural downward pressure on sigma (a root cause of entropy collapse); tanh-squash keeps all mass in (-1,1), smooth gradient, but REQUIRES a log-prob Jacobian correction (log(1-tanh^2)) and would make it a different algorithm (sect.8). (b) On gradient side there is NO inconsistency from clamping: the policy gradient uses log pi(raw a | o), i.e. the un-clamped sampled action, so "the clamped value isn't what's learned from" is false -- the raw sampled a IS what the update credits. The real issue is credit-assignment distortion: a=1.5 and a=3.5 both clamp to 1.0, so the out-of-range region [1, inf) is environmentally indistinguishable. This project stays on the standard external-clamp path and offsets its side-effects with the entropy bonus + per-dim min_std floor (see policy-head card), rather than switching to tanh.

WHY GAUSSIAN AT ALL (given "diverse" action distributions). Gaussian is the CONDITIONAL exploration distribution pi(a|o)=N(mu(o),sigma) for a single obs, NOT a claim that the marginal/optimal action distribution is unimodal. Diversity comes from mu(o) varying across situations; per-obs conditional Gaussian is enough. Rationale: (1) standard policy class ConstraintTRPO assumes (same family as SAC-tanh contrast in sect.8); (2) closed-form log-prob AND closed-form KL between two Gaussians -- the latter is what makes TRPO's trust-region cheap; a mixture/flow policy loses the analytic KL; (3) attitude stabilization is conditionally unimodal (the corrective torque/thrust direction is single per state), so multimodal policies buy little. Introducing a mixture/flow policy needs evidence of genuine multimodality first (rules/03: no generic ML upgrade without evidence).

SINGLE 8D VECTOR DRIVES TWO DYNAMICS vs MULTI-HEAD. The same 8D action feeds two physically different actuator paths (arm 2D delta-PD, thruster 6D TAM). This is NOT a problem per se: the policy emits only normalized [-1,1] commands and the env absorbs scale/dynamics differences (delta_scale 0.10 vs thrust_coeff 40; arm PD immediate vs thruster 1st-order filter lag, which the policy observes via the filtered thruster_state fed back into obs). The output layer is already 8 independent scalar heads (linear last layer, per-dim log_std); only the hidden trunk [256,128,64] is shared. So "single vs multi-head" really asks: should arm and thruster share the hidden trunk or split. CURRENT = shared trunk. The one real concern is gradient interference between arm-loss and thruster-loss on the shared trunk. BUT arm and thruster must COORDINATE for attitude control (thruster makes attitude, arm motion perturbs it via reaction/CoG), so they are strongly coupled -> shared trunk is the natural choice; splitting heads would cut this coupling at the representation level. Multi-head is only a win when the two subtasks are heterogeneous AND interfering, or when one subtask's gradient dominates the other. If the known per-axis learning imbalance (att_rp stalls while lin_vel learns) is caused by gradient conflict, a split could help IN THEORY -- but that must be MEASURED first (log cosine similarity of arm-loss vs thruster-loss gradients on the shared trunk; negative sustained = conflict; or separate-network ablation). Do not add multi-head without that evidence (rules/03).

ARM DELTA INTEGRATOR RESET semantics (why episode-only, not target-reached). _joint_pos_targets is a leak-free POSITION-COMMAND integrator (q_des += delta_scale*a, delta_scale=0.10, albc_env.py:578), NOT a PID error-integrator. Three writes only (sect.4.1): init to nominal (0.0, pi/2) at :278; += accumulation at :578; reset at :1315. The reset lives in _reset_action_buffers (:1311, called from reset at :1309) and sets q_des[env_ids] = MEASURED joint_pos (self._robot.data.joint_pos[...]), per-env. So accumulation is EPISODE-SCOPED: it grows unbounded WITHIN an episode (continuous-rotation joints, no wrap/clamp; joint1 cable protected by joint1_position_cost, not clamp) but is re-initialized at each episode reset, per-env. Resetting to the MEASURED position (not 0 / not nominal) is anti-bump init: PD torque ~ Kp*(q_des - q_measured), so starting q_des = q_measured makes the error 0 at reset -> no initial torque kick even under randomized initial pose. IMPORTANT CONCEPTUAL CORRECTION: a "reset the integrator when the joint REACHES the target, then compute the next target" scheme (waypoint-following / error-integrator picture) is WRONG for this structure and would BREAK control -- reaching the target is a HOLD state, not a reset trigger; resetting q_des to 0/nominal on reach would fling the arm back to origin. q_des is a live position command the policy steers via delta each step, not a one-shot waypoint. Only reset trigger that is valid = discarding cross-episode history = episode boundary.

ARM PD GAIN (Kp=100/Kd=3) basis, Dynamixel match, DR. Basis (albc.py:198-201 comments): THEORETICAL 2nd-order calc, NOT hardware measurement. Kp=100 -> "w_n=57.7 rad/s with J~0.15 kg*m^2" (natural freq from assumed inertia); Kd=3 -> "damping ratio ~0.7 (near critically damped)"; effort_limit_sim=13 Nm cites "motor stall torque 9.5 Nm" but NO motor model name in this file. Real hardware = Dynamixel XM430-W350 (sim-to-real.md:13,19), internal PID ~1kHz. NO code/values map sim gain to the real Dynamixel gain: sim-to-real.md:189-193 says direct mapping is impossible (register not SI, per-model conversion) and to use step-response system identification; that measurement is an unchecked TODO (sim-to-real.md:230-231). DR: YES. randomize_joint_gains (mdp/events.py:457-469) DIRECTLY uniform-samples ABSOLUTE ranges and OVERWRITES stiffness/damping (nominal 100/3 is NOT a scaled center, it's only the DR-off value). Default (config.py:126-127): stiffness [40,120], damping [0.5,5.0]. Hard (config.py:183-184, encoder training): stiffness [30,150], damping [0.3,7.0]. effort_limit is a SCALE [0.7,1.0] (config.py:129); joint friction static [0,0.03]/viscous [0,0.2]. Applied at env RESET only (albc_env.py:1410-1419, if rand_cfg.enable), NOT per-step; NOT a DORAEMON curriculum param (fixed range). KEY sim-to-real limit: gain DR randomizes the VALUE but cannot change the CONTROLLER FORM -- Dynamixel discrete PID (1kHz, integer registers, PWM saturation, firmware filters) vs PhysX continuous ImplicitActuator PD is a STRUCTURAL gap DR cannot cover (same class of "DR-uncoverable structural gap" as the thruster deadband+quadratic curve). No code addresses this structural gap yet; the three fixes (sysID gain-match / custom Dynamixel actuator model / control-rate match) all require the missing hardware step-response measurement. NOTE: docs/how-to/sim-to-real.md previously mis-recorded this DR as "+-20% ([80,120])" -- corrected 2026-07-02 to the absolute-range framing.

ADDENDUM 2026-07-02 (board field-test correction). The earlier claim in this card ("NO code/values map sim gain to the real Dynamixel gain") was SCOPE-LIMITED and is now CORRECTED by on-board measurement: the ROBOTIS SDK scan covered only /workspace/references/drivers/ (protocol library + Position-Control examples, genuinely no gain code), but the REAL joint driver lives in the agent-jetson repo (joint_angle_command.cpp + dynamixel_config.h, a nested independent git repo inside the vault, OUTSIDE /workspace) and DOES actively write Position gains to registers 84(P)/82(I)/80(D) = P=800/I=1/D=40 plus profile velocity. Also CORRECTED: the arm motor is Dynamixel XW540-T260 (board-confirmed: model# 1170 + 2.69 mA/LSB + IP68), NOT the "XM430-W350" this card and sim-to-real.md assumed. WHAT STILL HOLDS: register gains are integer/model-specific PWM-scaled, NOT SI, so they still cannot be dropped into ImplicitActuator Kp/Kd; the PID loop is still firmware-internal (~1 kHz, not source-visible); discrete-PID-with-I-term vs continuous-PhysX-PD is still a structural gap gain-matching cannot cover. So the card conclusion (step-response sysID is the real fix) is UNCHANGED -- only the "no gain code exists" and "XM430" specifics are corrected.
