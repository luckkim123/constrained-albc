# Actuator hardware identification: arm=XW540-T260 (board-measured), prior 'no PID-gain code' claim corrected, thrust-curve keep-off, step-response prereqs

Actuator HARDWARE identification session (2026-07-02, agent-jetson board field-test + code). Establishes what the albc/agent-jetson arm+thruster actuators actually ARE, and what a sim-to-real PD gap fix genuinely requires. Companion to the action-pipeline card [[action_pipeline_behavior_walk_through_two_clamps_raw_gaussian_vs]] (arm PD gain DR / basis), the thruster-curve card [[thruster_nonlinear_curve_t200_sim_to_real_off_by_default_deadban]] (nonlinear curve toggle), and the sim-identifiability card [[sim_hydro_nominal_is_analytical_not_measured_imu_pressure_can_an]] (thruster forces not separable on-vehicle).

ARM JOINT MOTOR = Dynamixel XW540-T260, CONFIRMED BY BOARD MEASUREMENT (not datasheet guess). 2 motors, ID 11 & 12. Confirmation = 3 concurring fingerprints: (a) model number register = 1170 (read live on-board via SDK ping after relay ON), (b) current unit 2.69 mA/LSB (from the project driver code), (c) IP68 waterproof (the deployment is underwater). All three match XW540-T260 and only it. Ruled out: MX-64 (model# 311, 3.36 mA/LSB) X, XW430-T333 (model# 1270) X, XW540-T140 (model# 1180) X. IMPORTANT DOC CORRECTION: docs/how-to/sim-to-real.md sect.1.1 table previously named the arm motor "Dynamixel XM430 / XM430-W350" -- that is WRONG; the real motor is XW540-T260 (the XW-series is the waterproof line; XM430 is not IP-rated). Fix the sim-to-real.md motor name accordingly.

PRIOR "PID gain = 0 code lines" CLAIM WAS WRONG -- corrected this session. An earlier Explore scan of ONLY /workspace/references/drivers/ (the vendored ROBOTIS SDK + its ROS examples) found no gain/operating-mode/goal-current register writes, and concluded "no PID-gain code anywhere." That conclusion was scope-limited and FALSE for the real robot: the actual joint driver lives in the agent-jetson repo (NOT in /workspace/references/drivers/), in joint_angle_command.cpp + dynamixel_config.h, and it DOES actively write Position gains to registers 84(P)/82(I)/80(D): P=800, I=1, D=40, and also sets profile velocity. LESSON (rules/03 "no absence from partial scan"): /workspace/references/drivers/ is a two-LAYER thing -- (1) vendored SDK = pure protocol/comm library (port, packet, CRC, generic register read/write; Position-Control-only examples using addr 64 TorqueEnable / 116 GoalPosition / 132 PresentPosition; NO gain code), and (2) the PROJECT driver in the agent-jetson repo that USES the SDK to write gains. Scanning only layer 1 says nothing about layer 2. The agent-jetson code is a NESTED independent git repo inside the vault (0_Project/in_progress/albc/code/agent-jetson/, own .git, excluded by vault .gitignore, NOT a submodule) -- it is real code in the proper code/ path, not tmp, but it is OUTSIDE /workspace so a /workspace grep never sees it.

WHAT STAYS TRUE despite that correction: the PID LOOP ALGORITHM itself is inside the motor firmware (~1 kHz, closed, not source-visible); the host only WRITES the gain setpoints into registers, it does not run the loop. And register gains (P=800/I=1/D=40, integer, model-specific PWM scaling) are NOT SI PD gains -- they cannot be dropped into the sim's ImplicitActuator Kp/Kd (Kp=100 N*m/rad, Kd=3 N*m*s/rad, SI). So knowing the model name + register gains does NOT by itself close the PD sim-to-real gap: (1) register->PWM->torque conversion coeff is model/firmware-specific and not published in SI; (2) even with it, discrete PID @1kHz WITH an I-term (I=1, nonzero) + PWM saturation + trapezoidal profile-velocity motion is a DIFFERENT CONTROLLER FORM than the sim's continuous PhysX PD (no I-term, no profile) -- a structural gap gain-matching cannot cover. sim-to-real.md sect.5.4 already said the register is not SI and step-response sysID is the practical route; this session confirms it empirically.

WHAT THE MEASUREMENT DID BUY (it is a PREREQUISITE, not the fix). The gap-closing route is step-response system identification (sim-to-real.md sect.5.1-5.3), and this session locked its preconditions: (a) measurement TARGET fixed = XW540-T260 (before this, MX-64 was still a candidate -> you didn't even know what to measure); (b) the gain CONDITION to reproduce = P=800/I=1/D=40 (must measure at the real operating gains, not altered ones, else the sim's target diverges from the deployed robot); (c) I=1 (nonzero) empirically CONFIRMS the structural continuous-PD-vs-discrete-PID gap exists; (d) profile-velocity is set -> real motion is trapezoidal-profiled, unlike the sim delta-integrator trajectory. The still-MISSING decisive piece is the DYNAMIC step-response trajectory (goal-step in, present-position logged over time) -- that is the target curve the sim Kp/Kd/delay_steps/velocity_limit get tuned against. It can be captured from the same relay-ON + SDK-ping setup by writing GoalPosition(116) as a step and logging PresentPosition(132) at >=200 Hz. FIELD-TEST BONUS re-confirmed: "relay OFF => motor unpowered (no response at any baud); relay ON => normal response" (matches the prior auto-memory field-test conclusion, now re-verified live).

THRUSTER PD/CURVE = NOT deploy-transparent, and NOT identifiable from code (distinct from fault). Decision this session: KEEP thrust curve OFF (enable_thrust_curve=False, the default; main has no curve at all) UNTIL a real command->thrust curve is bench-measured. Reason -- the curve is a PLANT MODEL (command->force map), NOT a DR-style perturbation. Contrast with fault/health: fault is a training perturbation, so an approximate model is still a net win and is irrelevant at deploy (no fault at deploy). The curve is the opposite: the policy learns a command strategy that PRESUMES the curve's force response, so at deploy the real thruster must reproduce that same curve or the policy is optimized for the WRONG plant -- an inaccurate curve can be WORSE than the linear default (a known gap) because it manufactures a NEW gap. And the current signed-square+deadband(0.075) has NO measured/literature backing (provenance unverified: not the same shape as the in-tree marinegym RPM-space piecewise-quadratic; EasyUUV paper absent from repo). So: curve ON pays off ONLY IF signed-square+deadband actually approximates the real T200; that is unverified, so leave it OFF and train/deploy on the linear plant. Re-enable path: bench-measure the real thruster command->thrust curve -> confirm/replace the shape in _thrust_command -> then train curve-ON. NOTE the deploy question "just deploy as before?" answers: only safe IF the real thruster naturally realizes the trained curve (path A), OR the deploy code explicitly re-applies the same _thrust_command (path B); "as before" (linear passthrough) is only correct when curve is OFF (which it is).

STEP-RESPONSE MEASUREMENT PROMPT (for a future agent-jetson board session). Safety-first: clear the arm's swept volume, relay ON (else no power), E-stop within reach, small amplitude first. Read the ACTUAL driver (joint_angle_command.cpp, dynamixel_config.h, vendored SDK) to reuse its port/baud(examples default 57600)/Protocol-2.0 API and the real position<->rad constant (do NOT invent addresses/constants). Measure at the REAL operating gains P=800/I=1/D=40 (read-verify regs 84/82/80 first; log profile velocity + operating mode); per joint (ID11 then ID12, never both at once). Step: GoalPosition(116)=q0+delta, delta ramped 5deg->15deg->30deg, both directions, 3 trials each; log PresentPosition(132) [+PresentVelocity 128 if available] at >=200 Hz from t=0 until fully settled (>=2 s). CSV schema: timestamp_s(0 at step), joint_id, goal_pos_rad, present_pos_rad, present_pos_tick, present_vel_rad_s, step_deg, direction, trial_idx. Plus measurement_meta.md: confirmed XW540-T260 + live model#, P/I/D + profile-vel + operating-mode reg reads, baud/protocol/port, achieved sampling rate, q0, anomalies. Finish: TorqueEnable OFF, relay OFF. NEVER change gains for the measurement (that moves the sim's target off the deployed robot).
